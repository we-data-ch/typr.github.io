Définir des task flows simples (90 points)

Concept clé​

Un ask flow représente le chemin qu'un utilisateur suit pour accomplir une tâche spécifique. C'est une séquence linéaire d'étapes sans variations.

Sructure :

  • Pint d'entrée → Actions → Décisions → Point de sortie
  • Fcus sur UNE tâche à la fois
  • Ps de choix alternatifs (contrairement au user flow)

Exercice pratique​

Cnsigne : Créez le task flow pour la tâche "Commander un smartphone samsung" and le site galaxus. Documentez chaque étape de manière séquentielle.

Lvrables attendus :

  • Dagramme du flux (formes standards : rectangles pour actions, losanges pour décisions)
  • Anotations des informations nécessaires à chaque étape
  • Ientification des points de friction potentiels

hecklist de fin d’exercice — Task Flow “Commander un smartphone Samsung sur Galaxus”​

1. Respect du concept de task flow​

  • Le flux couvre une seule tâche : commander un smartphone Samsung
  • Le flux est linéaire, sans chemins alternatifs ou scénarios optionnels
  • Le point d’entrée est clairement identifié
  • Le point de sortie est clairement identifié (commande confirmée)
  • Les décisions sont limitées au strict nécessaire (pas de user flow déguisé)

2. Complétude des étapes​

  • Toutes les étapes nécessaires à la commande sont présentes
  • Les étapes sont décrites dans un ordre strictement séquentiel
  • Aucune étape implicite n’est laissée à l’interprétation
  • Le flux commence avant l’action principale (ex. arrivée sur le site)
  • Le flux se termine après l’action finale (ex. confirmation de commande)

3. Diagramme du flux​

  • Un diagramme est fourni
  • Les rectangles sont utilisés pour les actions
  • Les losanges sont utilisés pour les décisions
  • Les flèches indiquent clairement le sens du parcours
  • Le diagramme est lisible et compréhensible sans explication orale

4. Annotations des informations​

  • Chaque étape contient les informations nécessaires à l’utilisateur
  • Les données demandées sont précisées (ex. adresse, moyen de paiement)
  • Les éléments visibles à l’écran sont mentionnés lorsque pertinent
  • Les actions système (validation, chargement, confirmation) sont notées

5. Points de friction​

  • Des points de friction potentiels sont identifiés
  • Chaque point de friction est associé à une étape précise
  • Les frictions sont réalistes (temps, complexité, ambiguïté, effort cognitif)
  • Il est clair pourquoi chaque point peut poser problème à l’utilisateur

6. Cohérence avec Galaxus​

  • Le flux est cohérent avec un site e-commerce réel comme Galaxus
  • Les termes utilisés correspondent à l’univers du e-commerce
  • Aucune fonctionnalité irréaliste ou inventée n’est introduite
